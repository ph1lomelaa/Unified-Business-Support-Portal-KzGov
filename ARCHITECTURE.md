# Архитектура проекта ЕППБ

## Цель

Документ описывает архитектуру MVP платформы «Единый портал поддержки бизнеса» (ЕППБ), включая состав компонентов, архитектурные слои, модель данных, интеграции, AI-модуль, функциональные сценарии и план развития.

## Executive Summary

ЕППБ — MVP цифрового «единого окна» для предпринимателей, которым доступны меры государственной и холдинговой поддержки дочерних организаций АО «Байтерек» (Даму, Банк развития Казахстана, Аграрная кредитная корпорация, KazakhExport, Отбасы банк и др.).

Цель решения — собрать разрозненные меры в единый каталог, помочь подобрать подходящую меру, подать заявку онлайн и отслеживать её статус.

Ключевые модули:

- Каталог услуг и AI-подбор меры поддержки
- No-code конструктор услуг и версионирование форм
- Подача многоэтапной заявки и личный кабинет
- AI-слой для навигации, помощи, генерации и проверки
- Интеграционная шина для mock- и будущих боевых интеграций
- Аналитика, карта проектов и база знаний

Технологический стек:

- Frontend: Next.js 16, React 19, TypeScript, SurveyJS
- Backend: FastAPI, SQLModel, Pydantic
- База данных: PostgreSQL 16 (Docker) / SQLite локально
- AI: OpenAI gpt-4o-mini (по умолчанию), опционально Anthropic Claude
- DevOps: Docker Compose

Публичный демо-домен: `https://eppb-baiterek-muslima.duckdns.org`

## 1. Обзор решения

Система построена из следующих логических слоёв:

1. браузерный фронтенд на Next.js с серверным BFF-прокси
2. backend-приложение на FastAPI
3. реляционное хранилище через SQLModel
4. AI-модуль над детерминированным ядром
5. интеграционная шина для внешних обменов

Браузер обращается к своему источнику: серверный слой Next.js через rewrite `/bff` проксирует запросы в backend. Это упрощает работу с cookie-сессией и CORS.

### 1.1 Состав компонентов

| Компонент | Технология | Назначение |
| --- | --- | --- |
| Frontend | Next.js 16, React 19, TypeScript | Публичный сайт, личный кабинет, админ-панель, BFF-прокси `/bff` |
| Backend | FastAPI, SQLModel, Pydantic | REST API каталога, заявок, интеграций, справочников, контента, аудита |
| База данных | PostgreSQL 16 (Docker) / SQLite (локально) | Хранение сущностей и JSON-колонок |
| AI-модуль | OpenAI gpt-4o-mini / Anthropic Claude | Навигатор, чат, генерация услуги, проверка заявки |
| Интеграционная шина | backend/app/integration | Унифицированные внешние вызовы, mock, rest, esb, oidc, eds |
| Хранилище файлов | backend/storage | Сгенерированные PDF-заявления и загруженные документы |

## 2. Быстрый старт

### Запуск через Docker Compose

В корне проекта есть `docker-compose.yml` с тремя сервисами: `postgres`, `backend`, `frontend`.

- `docker compose up --build`

При первом запуске на пустой базе backend автоматически seed-ит демо-данные: услуги, заявки, справочники, интеграции, отчёты, проекты.

Сервисы в локальной среде:

- `frontend`: `http://localhost:3007`
- `backend`: `http://localhost:8000`
- Swagger UI: `http://localhost:8000/api-docs`
- PostgreSQL: `5432` внутри сети Docker

LLM-ключи передаются через переменные окружения: `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `AI_PROVIDER`. Если ключ не задан, AI-функции работают на детерминированном фолбэке.

### Локальный запуск без Docker

Через `Makefile`:

- `make install` — создать виртуальное окружение и установить зависимости
- `make seed` — загрузить справочные данные и демо-услуги
- `make backend` — запустить FastAPI на `:8000`
- `make frontend` — запустить Next.js на `:3000`

### Демо-режим и вход

Страница `/login` предлагает три способа входа:

- `eGov mobile` — имитация OIDC-авторизации через государственный IDP
- `ЭЦП` — подпись серверного nonce через NCALayer, разбор сертификата на ИИН/БИН/ФИО
- `Демо-роли` — быстрый вход «Предприниматель», «Аналитик», «Администратор»

Демонстрационный БИН предпринимателя: `123456789012` (компания ТОО «AgroDala»). Под этим БИН уже есть тестовые заявки в разных статусах.

## 3. Архитектура и модель данных

Схема данных описана декларативно в `backend/app/models.py` через SQLModel. Имена полей совпадают с контракта TypeScript-фронтенда (camelCase).

Структурированные данные (`условия`, `схема формы`, `ответы`, `mock-наборы интеграций`) хранятся в JSON-колонках.

### 3.1 Ключевые сущности

- `Organization` — дочерняя организация холдинга: id, название, цвет, логотип.
- `Service` — услуга/мера поддержки: slug, категория, условия, документы, материалы, теги, статус, срок рассмотрения.
- `FormSchema` — версия формы заявки услуги: JSON-схема SurveyJS, активная версия, автор.
- `Application` / `ApplicationEvent` — заявка: ответы, расчёт, снимок схемы, PDF, файлы и лента событий.
- `Company` / `CompanyProfile` — профиль юрлица по БИН: ОКЭД, регион, руководитель, контакты.
- `IntegrationSystem` / `Operation` / `Contract` / `Call` — реестр интеграций, операции, контракты, журнал обменов.
- `Dictionary` / `DictionaryItem` — справочники (регионы, ОКЭД, категории) с иерархией и внешней синхронизацией.
- `Report` / `Project` / `Calculator` — no-code контент: аналитика, проекты для карты, калькуляторы.
- `StatusModel` / `StatusTransition` — настраиваемая модель статусов заявки и разрешённые переходы.
- `KnowledgeItem` / `NewsItem` / `FaqEntry` — база знаний, новости, FAQ.
- `AuditLog` / `Notification` / `User` — журнал действий, уведомления, учётные записи.
- `OfficialSource` / `OfficialImportRun` / `ImportedService` / `ExtractedEvidence` — внешние источники и импортированные данные.

### 3.2 Конструктор услуг

Услуга хранится в двух связанных таблицах:

- `Service` описывает карточку меры: категорию, описание, условия, документы, материалы, теги, срок рассмотрения, статус (`draft` / `published` / `archived`).
- `FormSchema` хранит форму заявки отдельно как JSON-схему SurveyJS.

Каждое сохранение формы создаёт новую версию. Флаг `isActive` отмечает текущую действующую версию.

При подаче заявки активная схема записывается в снимок (`schemaSnapshot`) внутри `Application`, чтобы заявка не «плыла» при изменениях формы.

Форма может быть многоэтапной: страница помечается свойством `stage: 2`, и модуль `schema_stages` разбивает схему на страницы первого и второго этапа.

- Первичная подача проходит по страницам первого этапа.
- Дополнительные сведения второго этапа заполняются позже из кабинета без изменения кода, только разметкой страниц.

Расчётные поля формы (`expression`) вычисляются на сервере безопасным вычислителем выражений `calc_eval`. Поддерживаются арифметика и функции `pow`, `min`, `max` без `eval`. Результат сохраняется в поле `calc` заявки и попадает в PDF.

### 3.3 AI-модуль

AI реализован как слой «AI поверх детерминированного ядра»: при наличии ключа LLM уточняет формулировку и предлагает варианты, а при отсутствии ключа, ошибке или таймауте система откатывается к офлайн-логике.

Форма ответа одинакова для frontend, поэтому интерфейс не различает источник. Каждый ответ помечается полем `source` (`ai` / `rules` / `fallback`).

Активный провайдер в текущем стенде — OpenAI `gpt-4o-mini`. Модуль `ai/client.py` содержит адаптер, приводящий вызовы к единому интерфейсу Anthropic Messages API; при заданном ключе Anthropic используется Claude.

Перед отправкой любого промпта персональные данные минимизируются: ИИН/БИН, e-mail и телефоны маскируются функцией `redact_personal_data`.

### 3.4 Точки применения AI

| Модуль | Файл | Описание |
| --- | --- | --- |
| Навигатор подбора меры | `ai/navigate.py` | По свободному описанию бизнеса возвращает до 3 услуг с обоснованием; кэширует результат; фолбэк — поиск по ключевым словам |
| Чат-консультант | `ai/chat.py` | Диалог с учётом истории и контекста базы (услуги, статьи, калькуляторы, FAQ); возвращает ответ + карточки- ссылки |
| Генерация услуги из текста | `ai/generate.py` | Из текста программы извлекает карточку услуги и черновик формы; результат валидируется, выполняется автоисправление, открывается как черновик |
| Подсказки полей конструктора | `ai/construct.py` | Предлагает набор полей и правила ветвления `visibleIf` с подтверждением аналитика |
| Аудит схемы формы | `ai/construct.py` | Ищет пропущенные идентификаторы, отсутствие загрузки документов, списки без вариантов, битые условия, некорректные формулы |

### 3.5 Механизм имитации внешних интеграций

Портал не обращается к внешним системам напрямую. Любой обмен проходит через единый вызов шины:

```python
bus.call(db, system_id, operation_code, payload, ...)
```

Шина `backend/app/integration/bus.py` находит конфигурацию системы и операции в базе, выбирает адаптер по полю `adapterType` и выполняет вызов.

Свойства шины:

- идемпотентность: при повторном вызове с тем же `idempotencyKey` возвращается предыдущий успешный ответ
- повторные попытки: `retryPolicy` (maxAttempts, backoffMs)
- журнал обменов (outbox): каждая операция сохраняется как `IntegrationCall` с статусом, попытками, задержкой и payload
- унифицированный ответ: `{ok, data, status, source, latencyMs, attempts, callId}`

Зарегистрированные адаптеры:

- `mock` — возвращает admin-редактируемый mock-набор или делегирует resolver
- `rest` — реальный HTTP-вызов `baseUrl + path` с подстановкой токенов
- `esb` — единая интеграционная шина для маршрутизации заявок в BPM дочерних организаций
- `oidc` — eGov IDP: имитация OIDC-логина и callback
- `eds` — НУЦ / ЭЦП: имитация CMS-подписи документа

Добавление новой интеграции существующего типа выполняется через админ-панель: новая система + операция + mock-данные без изменения кода. Замена mock-ответа на реальный endpoint не требует изменений в вызывающих потоках.

## 4. Функциональные модули

### 4.1 Каталог услуг и подбор меры поддержки

Каталог `/services` отдаёт опубликованные услуги с фильтрами по региону, виду деятельности, институту развития и размеру бизнеса.

Эндпоинты:

- `GET /api/v1/services` — список услуг
- `GET /api/v1/services/facets` — значения фильтров
- `GET /api/v1/services/{slug}` — карточка услуги и активная форма

Карточка услуги показывает условия, документы, материалы, FAQ и предварительную проверку пригодности.

AI-навигация на главной странице принимает описание бизнеса на свободном языке и возвращает до трёх подходящих услуг с обоснованием «почему подходит». Запрос сохраняется в сессии.

### 4.2 Конструктор услуг (админ-панель)

Конструктор `/admin/services/{id}` собирает услугу без программирования. Вкладки:

- `Карточка` — описание, условия, документы, материалы
- `Форма заявки` — визуальный сборщик SurveyJS с пресетами
- `Шаблон документа`
- `Публикация`

Сохранение формы создаёт новую версию; публикация переводит услугу в `published`.

AI-помощники:

- `Предложить поля`
- `Предложить ветвление`
- `Проверить форму на дыры`

Все предложения применяются только после явного подтверждения аналитика.

### 4.3 Подача заявки (мастер, многоэтапные услуги)

Мастер подачи `/services/{slug}/apply` реализует:

- создание черновика заявки `POST /api/v1/applications`
- автосохранение данных и расчётов `PATCH /api/v1/applications/{id}`
- проверка и подпись
- отправку заявки `POST /api/v1/applications/{id}/submit`

При отправке система:

- снимает снимок активной схемы
- вычисляет расчётные поля
- генерирует PDF-заявление
- создаёт статусное событие и уведомление
- маршрутизирует заявку в BPM дочерней организации через шину с идемпотентностью

Если услуга многоэтапная (`stage 2`), после первичной подачи заявка получает статус `нужны дополнительные сведения`. Заявитель дозаполняет данные из кабинета `POST /api/v1/applications/{id}/stage2`, после чего PDF перегенерируется и заявка уходит на рассмотрение.

### 4.4 Личный кабинет пользователя

Кабинет `/cabinet` показывает заявки предпринимателя с разделением на требующие действия и активные.

Разделы:

- `Мои заявки`
- `Документы` (сгенерированные PDF и загруженные файлы)
- `Уведомления`
- `История действий`
- `Профиль`

Карточка заявки `/cabinet/applications/{id}` отображает ленту статусов, прогресс SLA, данные компании и панель второго этапа при необходимости.

### 4.5 AI-компонент

Плавающий чат-консультант доступен на публичных страницах. Он отвечает на языке пользователя, учитывает историю диалога и контекст базы портала (услуги, статьи, калькуляторы, FAQ), возвращает карточки-ссылки и подсказки следующих вопросов.

### 4.6 Аналитика дочерних организаций

Раздел `/reports` сочетает дашборд и каталог материалов. Показывает ключевые показатели:

- объём финансирования
- число проектов
- созданные рабочие места
- финансирование по годам
- топ регионов
- распределение по отраслям и статусам
- сравнение дочерних организаций

Данные считаются по таблице проектов. Клиентские фильтры дают пересчёт в реальном времени.

### 4.7 Карта проектов

Карта `/map` отображает профинансированные проекты по регионам Казахстана.

Эндпоинты:

- `GET /api/v1/map/projects`
- `GET /api/v1/map/regions`

В демо-данных отображено 190 проектов с координатами, отраслью, суммой и статусом.

### 4.8 База знаний и полезные материалы

Раздел `/knowledge` содержит статьи, чек-листы, гайды и шаблоны (`KnowledgeItem`) в облегчённом markdown. Отдельный раздел `/calculators` предоставляет рабочие инструменты, где админ задаёт входные поля и формулу, а frontend рендерит расчёт с графиком.

Формула рассчитывается на сервере тем же безопасным вычислителем, что и расчётные поля формы.

### 4.9 Административные разделы

Админ-панель `/admin` включает:

- реестр услуг
- очередь заявок оператора и смену статусов
- консоль интеграций
- справочники
- импорт мер из внешних источников
- управление отчётами, проектами, калькуляторами, статусами/маршрутами, базой знаний
- журнал аудита действий

## 6. Имитация внешних интеграций

Все внешние обмены проходят через интеграционную шину и настраиваются в админ-консоли «Интеграции и шина».

В реестре зарегистрировано несколько систем и операций. Каждое действие записывается и может быть проверено через тестовый вызов.

### Имитации в MVP

| Система | Адаптер | Как работает | Что требуется для продакшена |
| --- | --- | --- | --- |
| Вход через eGov | `oidc` | Поток `/egov/start` → `/egov/callback` через шину; адаптер возвращает `authenticated`, `subject` (ИИН), уровень доверия | Подключить продуктивный `client_id` и OIDC discovery после аккредитации стенда |
| Подпись ЭЦП | `eds` | Имитация CMS-подписи документа, разбирается сертификат NCALayer | Подключить проверку CMS-цепочки и timestamp против корня НУЦ |
| Профиль компании по БИН | `mock` (`gbd-ul`) | Resolver `gbd.company` читает БИН из сид-таблицы `Company` | Заменить mock на ESB-endpoint ГБД ЮЛ, сохранив контракт |
| BPM дочек | `esb` | Заявка маршрутизируется в BPM через ЕИШ: чеканится внешний id, идемпотентность, запись в outbox | Согласовать справочник кодов услуг/статусов и подключить боевой ЕИШ/BPM |
| Справочники | `mock` (`nsi-registry`) | Resolver `nsi.dictionary` отдаёт элементы справочника; админ-синхронизация тянет их в `DictionaryItem` | Подключить продуктивный реестр НСИ |
| Уведомления | `notifications` (planned) | В MVP уведомления пишутся в таблицу `Notification` и показываются в кабинете | Подключить провайдера рассылок и таблицу delivery receipts |

## 7. Известные ограничения MVP

Текущие упрощения MVP:

- Внешние государственные и холдинговые системы работают как имитации через адаптеры шины и не подключены к боевым сервисам.
- Вход по ЭЦП разбирает субъект сертификата NCALayer, но полная проверка CMS-цепочки и метки времени против корня НУЦ не реализована.
- Активный LLM-провайдер стенда — OpenAI gpt-4o-mini; при отсутствии ключа AI-функции переходят на детерминированный фолбэк.
- Сессия реализована через подписанную cookie с демонстрационным секретом; роли в демо-режиме выбираются кнопкой без реального провайдера идентификации.
- По умолчанию локально используется SQLite; продуктивная конфигурация в Docker — PostgreSQL. Миграции схемы не оформлены отдельным инструментом (таблицы создаются из моделей, данные сидируются скриптами).
- Ответ чат-консультанта не потоковый, используется одиночный запрос с увеличенным таймаутом.
- Планировщик импорта (apscheduler) опционален: если библиотека не установлена, автоматический запуск отключается, ручной импорт остаётся доступен.
- Реальный HTTP-запуск импорта поддержан для части источников; остальные источники возвращают ошибку до подключения адаптера.
- Уведомления формируются внутри портала; внешняя доставка по Email/SMS/WhatsApp не подключена.

## 8. Дальнейшее развитие

Планы развития:

- Подключение боевых интеграций поверх существующих контрактов шины: eGov IDP (OIDC), ГБД ЮЛ, НУЦ/ЭЦП, ЕИШ и BPM дочерних организаций, реестр НСИ.
- Полная криптографическая проверка ЭЦП (CMS-цепочка, timestamp, статус сертификата) против корня НУЦ.
- Внешняя доставка уведомлений (Email/SMS/WhatsApp) с журналом доставки.
- Инструмент миграций схемы БД и разделение сид-данных и справочных констант.
- Потоковая передача ответов чата (SSE).
- Расширение базы знаний и калькуляторов.
- Расширение аналитики и выгрузок, ролевая модель оператора дочерней организации.

## Глоссарий

- MVP — Minimum Viable Product
- BFF — Backend for Frontend, серверный слой фронтенда
- Интеграционная шина — единая точка всех внешних обменов
- Адаптер — подключаемый модуль, превращающий операцию в конкретный протокол
- Идемпотентность — повторный вызов с тем же ключом даёт тот же результат
- Outbox — журнал исходящих / входящих обменов
- Webhook / callback — входящий вызов от внешней системы
- OIDC — OpenID Connect
- ЭЦП / CMS — электронная цифровая подпись / формат CMS/PKCS7
- NCALayer — клиентское ПО НУЦ для ЭЦП в браузере
- ЕИШ — единая интеграционная шина холдинга
- BPM — Business Process Management дочерней организации
- ГБД ЮЛ — государственная база данных юридических лиц
- НСИ — нормативно-справочная информация
- ОКЭД — Общий классификатор видов экономической деятельности
- БИН / ИИН — бизнес- и индивидуальный идентификатор налогоплательщика
